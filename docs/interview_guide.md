# 智能工单 RAG 系统 — 面试深度指南

> 预期时长 40-60 分钟，覆盖架构设计、技术选型、工程实践和场景追问

---

## 一、项目概述（2 分钟自我介绍）

**一句话**：面向医院设备科的医疗工单 RAG 问答系统，用 Milvus 向量检索 + 千问 LLM 生成故障排查建议。

**规模**：22 个 chunk / 11 张工单（演示），架构支持百万级扩展。

**我个人贡献的亮点**：
- 三模式检索（语义 + 关键词 + 混合加权融合）
- 三级对话记忆（Redis 短期 → Kafka 管道 → MySQL 长期 + LLM 摘要）
- 语义缓存（embedding 余弦匹配，9x 加速）
- 三层限流（Token Bucket + Sliding Window，Redis Lua 原子化）
- Prompt 工程（Few-shot + CoT 推理链 + JSON Schema 双输出）

**追问①：为什么不用 LangChain 全套？**

我只在 ingestion 层用了 LangChain 的 `RecursiveCharacterTextSplitter`（成熟稳定，懒得重写），其他部分全部自己封装——LLM 调用、检索引擎、记忆管理都是原生实现。原因：LangChain 的链式调用抽象层太厚，调试困难；它的 Memory 模块不支持我设计的 Kafka 异步管道；它的 Retrieval 抽象不适合 Milvus 混合检索的原生 API。

## 三、检索系统深潜（重点考区）

### 3.1 为什么三种模式？

| 模式 | 原理 | 优势 | 劣势 |
|------|------|------|------|
| semantic | 1536d 稠密向量做 ANN | 同义词/语义相近能召回 | 精确术语匹配弱 |
| keyword | BM25 稀疏向量倒排索引 | 工单号/型号精确命中 | 不理解语义 |
| hybrid | 两路并行 → WeightedRanker 加权融合 | **互补盲区** | 多一次编码开销 |

**追问②：hybrid 的加权融合是怎么实现的？**

Milvus 原生支持 `hybrid_search` + `WeightedRanker`。内部机制：
1. 两个 `AnnSearchRequest` 各检索 `top_k * 2` 条候选
2. `WeightedRanker(dense_weight, sparse_weight)` 对两路分数做线性加权
3. 融合后取 `top_k`

默认 0.5:0.5，实际场景可以根据查询类型动态调整——如果用户输入的是工单号 `GD-2026-03001`，应该调高 sparse 权重；如果是自然语言描述，应该调高 dense 权重。

**追问③：BM25 的 jieba 分词怎么处理医疗术语？**

BM25 默认的 jieba 分词会把"血液透析机"切成"血液/透析/机"，丢失专业语义。我在初始化时用 `jieba.add_word()` 预加载了 27 个医疗领域自定义词汇（"电导率"、"血液透析机"、"CT扫描"等），强制不拆分。

**追问④：IVF_FLAT vs HNSW 怎么选？**

选了 IVF_FLAT：
- 22 条数据时：IVF_FLAT 和 HNSW 性能无差异
- 百万级时：HNSW 更快但内存占用高（图结构需要全在内存），IVF_FLAT 更省内存
- 医疗场景更看重召回率而非极致延迟 → IVF_FLAT 的 `nprobe` 可调，精度可控

如果未来数据量上千万且对延迟敏感（<10ms），会切到 HNSW。

### 3.2 重排序（两阶段检索）

```
Stage 1 粗召回: top_k × 3 条 (双编码器，快)
Stage 2 精排:   qwen3-rerank 交叉编码器 (慢但准)
```

**追问⑤：双编码器 vs 交叉编码器的本质区别？**

双编码器（embedding 模型）：Query 和 Doc 分别编码为向量 → 余弦/内积计算相似度。**Q 和 D 没有交互**，所以可以做 ANN 索引（速度快）。

交叉编码器（reranker）：Q 和 D 拼接后一起送入 Transformer，输出一个相关性分数。**Q 和 D 有全注意力交互**，更精准，但每条 (Q,D) 对都要重新推理（速度慢）。

所以用了"粗召回（双编码器，快）→ 精排（交叉编码器，准）"的两阶段策略。

**追问⑥：reranker 挂了怎么办？**

`Reranker.rerank()` 的 try-catch 里，API 失败时直接按原始向量分数降序取 top_k，降级但不中断服务。

## 四、Chunking 策略（细节考区）

```python
separators = [
    "\n════",    # 工单之间（最优先）
    "\n【",      # 字段标签如【故障现象】
    "\nStep",    # 排查步骤
    "\n\n",      # 段落边界
    "。",        # 中文句号
    " ",         # 空格兜底
]
```

**追问⑦：为什么不用固定大小分块？**

医疗工单有固定结构（编号→故障现象→排查→方案→结果），固定大小会把一个逻辑完整的故障分析切成两半。递归字符分割用自定义分隔符优先级，优先在工单边界和字段标签处断开，保证**每个 chunk 是一个语义完整的片段**。

**追问⑧：ticket_id 是怎么继承到每个 chunk 的？**

工单编号只在每个工单的开头出现一次。分割后第二个 chunk 起就不含编号了。`chunker.py` 里维护了一个 `_last_ticket_id` 变量：遍历 chunks 时如果当前 chunk 匹配到新 ticket_id 就更新，否则继承上一个。`device_type` 同理。

## 五、Prompt 工程（面试高频）

### 5.1 System Prompt 设计

```
角色定位 → 职责范围 → 思维链推理框架(5步) → 安全守则(4条) → 回答要求
```

**追问⑨：CoT 在这里到底有什么用？**

医疗故障排查是典型的**鉴别诊断**场景——同一症状（"图像模糊"）可能来自多个子系统（探测器/球管/重建算法）。CoT 引导模型：
1. 拆解故障现象
2. 列举可能原因（排优先级）
3. 对照工单证据逐一排除/确认
4. 形成最终判断

没有 CoT 时模型容易跳步，直接给出结论而缺少推理过程，导致建议不够具体。

**追问⑩：JSON Schema 输出怎么保证格式正确？**

Prompt 里给出了完整的 JSON Schema 定义 + 完整示例。但 Prompt 约束是"软约束"——如果模型返回了非 JSON 文本，`st.json()` 会捕获 `JSONDecodeError`，回退到原始文本展示。生产环境建议加一层 Pydantic 校验 + 重试机制。

### 5.2 Few-shot 示例为什么只放了一个？

一个高质量、关联性强的示例比多个通用示例效果好。示例太长会吃掉 context window（工单 chunks 已经占了很多 token）。而且在医疗场景，不同故障类型差异大，通用示例可能产生误导。

## 六、三级记忆系统（架构考区）

### 6.1 为什么是三级？

```
短期(Redis) → 毫秒级读写，存最近 20 条
中期(LLM摘要) → 每 5 轮压缩，控制 token 增长
长期(MySQL) → 永久存储，支持历史回溯
```

**追问⑪：为什么 Redis 和 MySQL 之间要加 Kafka？**

三个原因：
1. **解耦写入路径**：用户请求的响应时间不包含 MySQL 写入延迟
2. **削峰填谷**：高并发时消息堆积在 Kafka，Consumer 按自己节奏消费
3. **可靠性**：MySQL 挂了不丢数据，Kafka 堆积到 MySQL 恢复后继续消费

Kafka 不可用时，`MemoryManager` 跳过 Kafka 直接写 MySQL，保证记忆不丢。

**追问⑫：摘要什么时候触发？每 5 轮这个数字怎么定的？**

经验值。太频繁（每轮）→ LLM 调用太多次；太稀疏（每 10 轮）→ 摘要生成的间隔太长，中间会丢失上下文。5 轮约等于用户围绕一个故障的完整排查闭环。

**追问⑬：记忆是怎么注入 Prompt 的？**

```python
def build_memory_prompt(session_id):
    ctx = get_context(session_id)
    # → {"messages": [...], "summary": "前5轮讨论CT伪影..."}
    
    parts = []
    if ctx["summary"]:
        parts.append(f"【历史对话摘要】\n{ctx['summary']}")
    if ctx["messages"]:
        parts.append("【最近对话】\n" + 格式化)
    
    return 拼接后拼到 user_message 前面
```

摘要放在前面让模型先建立全局认知，再给最近对话的细节。每条消息截断到 300 字符，防止 Prompt 过长。

### 6.2 降级策略

| 组件挂了 | 行为 |
|---------|------|
| Redis | 跳过短期记忆，传入空的 memory context |
| Kafka | Producer 不可用 → 直接写 MySQL |
| MySQL | Kafka Consumer 写入失败 → 指数退避重试 3 次（0.5→1→2→4s） |
| 全挂 | RAG 正常问答，只是没有对话记忆 |

**追问⑭：记忆系统是必需的还是可选的？**

可选的。`session_id` 传了就启用记忆，不传就是无状态问答。这个设计让同一个 API 既能做单次问答也能做多轮对话。

## 七、语义缓存（细节考区）

### 7.1 核心机制

```
question → embedding(1536d) → 与 Redis List 中每条缓存的 embedding 做余弦相似度
→ max_score ≥ 0.95? → 命中(跳过LLM) / 未命中(走完整RAG → 存回缓存)
```

**追问⑮：为什么阈值是 0.95 而不是 0.85？**

宁可漏判（多调一次 LLM，成本可接受），不能误判（返回不相关答案，用户体验差）。医疗场景对准确性要求更高——"CT伪影"和"MRI伪影"的 embedding 余弦可能高达 0.9+，但它们是完全不同的设备。0.95 足够保守。

**追问⑯：为什么不用 Milvus 做缓存检索？**

当前 100 条缓存，O(n) 余弦遍历 ~5ms，够用。如果扩展到万级，确实应该把 embedding 索引迁到 Milvus 做 ANN（项目已有 Milvus，不需要新组件）。但当前保持简单——缓存和检索引擎解耦，Redis 挂了不影响检索。

**追问⑰：缓存会过期吗？知识库更新了缓存怎么办？**

TTL 1 小时整批过期。缺点是新旧条目同生共死。生产版应该：每条记录独立 TTL + 关联知识库版本号，检测到新版本时批量失效。

### 7.2 实测效果

```
第 1 次: 24014ms（完整 RAG）
第 2 次: 2721ms（缓存命中，只有 embedding）
加速比: 9x
```

## 八、限流与配额（系统设计考区）

### 8.1 为什么三层？

```
全局限流 (Token Bucket) → 保护整个服务不被单一流量高峰打垮
用户限流 (Token Bucket) → 保证公平性，一个用户不能占满配额
LLM 预算 (Sliding Window) → 保护钱包，单用户 token 消耗有上限
```

**追问⑱：Token Bucket vs 固定窗口 vs 滑动窗口怎么选？**

| 算法 | 适用场景 | 当前用途 |
|------|---------|---------|
| Token Bucket | QPS 限流（允许突发） | 全局 + 用户 QPS |
| Sliding Window (Sorted Set) | 大时间窗口精确统计 | LLM 预算（1小时窗口） |

不用固定窗口是因为**边界突发问题**：59 秒→00 秒瞬间，两个窗口交界处可能出现 2x 瞬时流量。Token Bucket 天然平滑。

**追问⑲：Redis Lua 脚本在这里的具体作用？**

Token Bucket 的操作不是原子的——`GET tokens → 计算 elapsed → 判断 → SET tokens` 如果不用 Lua，在并发下会出现竞态条件（两个请求同时读到 `tokens=1`，都以为自己能通过）。

Lua 脚本在 Redis 里是原子执行的——整个 `HGET → refill → 判断 → HMSET → EXPIRE` 过程不会被其他命令插入，天然线程安全。

**追问⑳：contextvars 是什么？为什么用它传 user_id？**

`contextvars.ContextVar` 是 Python 3.7 标准库，类似 thread-local 但**协程安全**。

问题：API 层（ask.py）知道 user_id，LLMClient 需要 user_id 上报成本，中间隔着 RAGChain。不想改所有函数签名（`generate(user_id=...)`）。

方案：API 入口 `set_current_user(id)` → LLMClient 内部 `get_current_user()`。每个 asyncio.Task 有独立上下文副本，协程间互不干扰。

### 8.2 容错

Redis 不可用 → `RateLimiter.__init__` 里 `ping` 失败 → 自动退回 `InMemoryTokenBucket`（`threading.Lock` + `time.monotonic()`）。损失分布式一致性，单进程够用。

成本上报失败 → `_report_cost()` 全 try-catch，静默忽略。限流是**增强**不是**阻塞**。

## 九、技术选型对比（必考题）

### 9.1 为什么 Milvus 而不是 Pinecone/Weaviate/Qdrant？

| 维度 | Milvus | Pinecone | Weaviate |
|------|--------|----------|----------|
| 部署 | 自托管 Docker | 仅 SaaS | 自托管/SaaS |
| 混合检索 | **原生 WeightedRanker** | 需手动合并 | hybrid search |
| 稀疏向量 | **原生 SPARSE_FLOAT_VECTOR** | 不支持 | 有限支持 |
| 成本 | 免费 | 按量付费 | 免费 |

选 Milvus 的核心原因：**原生支持稠密+稀疏混合检索**，不需要维护两套系统。而且医疗数据不适合上 SaaS（数据合规）。

### 9.2 为什么 Qwen-Max 而不是 GPT-4？

1. DashScope 国内访问稳定，不翻墙
2. 中文医疗场景 Qwen 表现不错
3. 成本更低（约 GPT-4 的 1/10）
4. 代码里 LLMClient 抽象了接口，换模型只需改 config.yml

### 9.3 为什么 Embedding 不用本地模型（如 BGE）？

本地部署 BGE-M3 可以实现 1024d 嵌入 + 稀疏向量一体化，但需要 GPU 推理资源。当前用 DashScope API 零运维成本，适合快速验证。如果以后日调用量超过 10 万次，API 成本高于 GPU 服务器，再考虑切本地。

## 十、场景追问（压力测试）

### Q: 知识库从 22 条涨到 22 万条，怎么保证检索质量？

1. **Chunking**：当前 500 字符可能太大 → 调小到 200-300，增加粒度
2. **索引**：IVF_FLAT `nlist` 从 1024 调到 4096（√N * 4）
3. **粗召回 + 精排**：多召回 20 条，用 reranker 精排到 5 条
4. **元数据预过滤**：先按设备类型过滤，再在子集里检索
5. **评估**：引入 Recall@K、MRR 等指标持续监控

### Q: 用户问"上次那个问题怎么解决"，怎么处理指代消解？

当前系统不处理指代消解——这是一个待做的改进。方案：在查询预处理阶段，如果检测到指代词（"那个"、"它"、"上次"），用 LLM + 最近对话历史做**指代消解改写**，把模糊指代替换为具体实体，再送入检索。

### Q: LLM 幻觉怎么控制？

1. **System Prompt 安全守则**：明确要求引用工单编号、无证据不说
2. **无结果不调 LLM**：`chunks` 为空直接返回 fallback，不给模型"编造"的机会
3. **低温度**：`temperature=0.1`，减少随机性
4. **未来可加**：检索结果和生成答案的**事实一致性校验**（NLI 模型判断答案是否被检索结果支撑）

### Q: 系统怎么监控和观测？

当前只有 Python logging + FastAPI 请求耗时中间件。生产版应该加：
- **Prometheus metrics**：QPS、检索延迟 P50/P99、LLM token 消耗、缓存命中率
- **Grafana 仪表盘**：实时监控面板
- **告警**：LLM API 连续失败 N 次 → 钉钉/企微通知

### Q: 多租户怎么做？

当前按 `session_id` 隔离对话记忆，但没有租户级数据隔离。多租户版：
1. Milvus Collection 增加 `tenant_id` 字段（标量索引）
2. 所有检索请求强制带 `tenant_id` 过滤
3. 记忆系统 Redis Key 前缀加 `tenant:{id}:`
4. 限流按 tenant 维度做配额管理

## 十一、自我反思（面试官爱问"如果能重来"）

**做得好的**：
- 混合检索 + 重排序的精度-效率平衡
- 三级记忆的降级设计，每个组件都可挂
- 语义缓存的 ROI（20 行核心代码，9x 加速）
- 限流的 Redis→内存自动 fallback

**可以改进的**：
1. **缺少检索评估** — 没有 Recall@K / MRR 指标，检索质量靠人工感觉
2. **Prompt 太长** — 光 System Prompt 就有约 500 token，加上 Few-shot 示例和检索结果，经常逼近 context window
3. **没有 A/B 测试框架** — 改 Prompt 或检索权重后无法量化对比
4. **Chunking 过于简单** — 纯字符分割，没有利用工单的结构化字段做更好的语义分块
5. **缓存策略粗糙** — 整 Key TTL，新老条目同生共死

## 十二、一句话总结

> **一个面向医疗工单场景的 RAG 系统，核心贡献是混合检索 + 三级记忆 + 语义缓存 + 限流配额四层架构，全链路降级设计保证任一组件故障不影响核心问答。**
